# サービス定義

---

## サービス一覧

| # | サービス | 責務 | トリガー |
|---|---------|------|---------|
| S1 | DataIngestionService | Slackからデータを取得・正規化・保存 | Events API（リアルタイム）+ EventBridge（バッチ） |
| S2 | JudgmentService | チャットデータを判定し、3分類＋理由を生成 | S1完了後（リアルタイム簡易）+ EventBridge（バッチ詳細） |
| S3 | ScoreService | ダメ度スコアを算出・保存 | S2完了後（バッチ） |
| S4 | ReportService | 定時レスポンスを生成・配信 | EventBridge（毎日18:00 JST） |
| S5 | ActionService | 行動記録・メンション追跡・結果判定 | ユーザー操作 + EventBridge（24h×7日メンションチェック + 48hリマインド） |
| S6 | DashboardService | フロントエンドにデータを提供 | ユーザーリクエスト（オンデマンド） |

---

## サービス詳細

### S1: DataIngestionService

```
トリガー:
  - Slack Events API（message イベント）→ リアルタイム取得
  - EventBridge（毎日17:00）→ バッチ取得（当日分の全データ）
  - ユーザー操作 → エクスポートファイルアップロード

処理フロー:
  1. Slackからメッセージデータを取得
  2. 正規化（チャンネルID、ユーザーID、タイムスタンプ、本文、リアクション数、スレッド返信数、メンション有無）
  3. DynamoDBに保存
  4. JudgmentServiceに判定リクエストを送信

出力:
  - 正規化されたチャットデータ（DynamoDB）
```

---

### S2: JudgmentService

```
トリガー:
  - S1完了後（リアルタイム簡易判定）
  - EventBridge（毎日17:30）→ バッチ詳細判定

処理フロー（リアルタイム簡易）:
  1. メタデータのみで判定（メンション有無、時間帯、チャンネル統計）
  2. 明確なケースのみ判定確定（時間外発言→偽マジメ、メンション0チャンネル→神サボリ）
  3. 曖昧なケースは「保留」としてバッチに回す

処理フロー（バッチ詳細）:
  1. 保留分 + 当日全データを対象にBedrock分析
  2. 本文AI分析（質問/依頼/期限の有無、報告系判定）
  3. 3分類判定（神サボリ/罪サボリ/偽マジメ）
  4. 判定理由の生成（Bedrock）
  5. DynamoDBに判定結果を保存

出力:
  - 判定結果（分類、理由、確信度）
```

---

### S3: ScoreService

```
トリガー:
  - S2バッチ完了後（毎日17:45頃）

処理フロー:
  1. 当日の判定結果を集計
  2. ダメ度スコアを算出: (神サボリ件数 / 全判定件数) × 100 - (罪サボリ件数 × 10)
  3. 創出時間を算出（神サボリ1件あたり平均15分として概算）
  4. 日次スコア履歴に追加
  5. 週次トレンドを更新

出力:
  - 日次ダメ度スコア
  - 累計創出時間
  - トレンドデータ
```

---

### S4: ReportService

```
トリガー:
  - EventBridge（毎日18:00 JST）

処理フロー:
  1. S2の判定結果とS3のスコアを取得
  2. レポートテンプレートにデータを埋め込み
  3. Bedrockで「一言コメント」と「学びポイント」を生成
  4. 時間外アラートを生成（18:00以降の発言を検出）
  5. Slack Webhookでユーザーにdm送信

出力:
  - Slack DMメッセージ（マジサボレポート）
```

---

### S5: ActionService

```
トリガー:
  - ユーザー操作（ミュート実行、問題報告）
  - EventBridge（24時間ごと）→ メンションチェック（1週間継続）
  - EventBridge（48時間後）→ リマインド送信

処理フロー:
  1. 行動を記録（タイプ、対象チャンネル、タイムスタンプ）
  2. 48時間後: 定時レスポンスで「問題ありませんでしたか？」リマインド
  3. 24時間ごと: 対象チャンネルで自分へのメンション有無をチェック
     - メンションなし → 「問題なし」継続
     - メンションあり → 「呼ばれてます」通知 + 罪サボリリスク記録
  4. 1週間経過: 最終判定
     - メンション0回 → 「サボり成功」確定
     - メンションあった → 結果に応じて判定修正
  5. 結果をJudgmentServiceにフィードバック（次回判定精度向上）

出力:
  - 行動記録（DynamoDB）
  - メンションチェック結果
  - 判定精度フィードバック
```

---

### S6: DashboardService

```
トリガー:
  - ユーザーリクエスト（API Gateway経由）

提供API:
  GET /dashboard          → ダメ度スコア、サマリー、今日の判定件数
  GET /judgments          → 判定一覧（フィルタ: 日付、分類）
  GET /trends            → トレンドデータ（日次/週次）
  GET /channels          → チャンネル別判定結果
  POST /actions          → 行動記録
  POST /actions/:id/report → 問題報告
  GET /settings          → Slack連携状態
  POST /settings/slack   → Slack連携設定
```

## サービスオーケストレーション（時系列）

```
17:00  EventBridge → S1 DataIngestionService（バッチ取得）
         ↓
17:30  EventBridge → S2 JudgmentService（バッチ詳細判定）
         ↓
17:45  S2完了 → S3 ScoreService（スコア算出）
         ↓
18:00  EventBridge → S4 ReportService（レポート生成・配信）
         ↓
       Slack DM → ユーザーに「今日のマジサボレポート」が届く

[日中リアルタイム]
  Slack Events API → S1 → S2（簡易判定のみ）→ DynamoDB
  ユーザー操作 → S5 ActionService → DynamoDB
  ユーザーがダッシュボードを開く → S6 DashboardService → レスポンス
```
